# Security Policy

`apohara-compliance` is a deterministic, offline compliance/agentic-security scanner.
This document covers how to report a vulnerability and how to verify the integrity of a
release.

## Reporting a vulnerability

Please report security issues **privately**, not via public issues.

- Preferred: open a [GitHub Security Advisory](https://github.com/SuarezPM/apohara-compliance/security/advisories/new)
  on this repository (private to maintainers until coordinated disclosure).
- We aim to acknowledge a report within **7 days** and to agree on a disclosure timeline
  with the reporter.

Please include: affected version (`apohara-compliance-scanner --version`), reproduction
steps, and impact. Coordinated disclosure is appreciated — we will credit reporters who
ask to be credited.

## Supported versions

| Version | Supported |
|---------|-----------|
| 1.0.x   | ✅ security fixes |
| < 1.0   | ❌ pre-release, not supported |

## Threat model (scope)

The scanner is intentionally small and isolated, which bounds its attack surface:

- **Offline and deterministic.** It makes **no network calls**, uses **no API keys**, and
  collects **no telemetry**. A CI guard (`scripts/verify.sh`) asserts the build pulls in no
  outbound-HTTP dependencies. Two runs over the same input produce the same output.
- **Untrusted inputs, parsed tolerantly.** Inputs are agent session transcripts (NDJSON)
  and repository trees, which are treated as untrusted. Malformed input is skipped, not
  trusted — the parser is designed to degrade gracefully rather than panic.
- **Advisory output only.** Every finding is a `CANDIDATE` for human review (SARIF
  `note`/`warning`, never `error`). The tool does not gate, block, or modify anything by
  itself; it cannot make a project "compliant" or "non-compliant".
- **Primary supply-chain surface = the distributed artifacts.** The most security-relevant
  surface is the published crate and the release binaries — i.e. the same agent
  supply-chain risk the scanner itself flags (`AGT-MIS-*`). Release integrity is addressed
  below.

Out of scope: the tool does not execute, sandbox, or remediate the code it scans; it reads
and maps. It is **not** a certification, an audit, or legal advice.

## Release integrity

Every tagged release is built and signed by the GitHub Actions release workflow
([`.github/workflows/release.yml`](.github/workflows/release.yml)) for all four targets
(linux x86_64, macOS arm64, macOS x86_64, windows x86_64). Each release binary ships with:

- a **build-provenance attestation** generated by
  [`actions/attest-build-provenance`](https://github.com/actions/attest-build-provenance)
  — a **SLSA-provenance-format** attestation that records *what* built the binary, *from
  which* source, *in which* workflow. (This is the SLSA provenance **format**; this project
  does **not** claim a specific SLSA *Level*.)
- a **cosign keyless signature** (`.sig` + `.pem`), bound via OIDC to the release workflow
  identity — no long-lived signing key exists.
- an entry in **`SHA256SUMS`**.

The release job **aborts** if any of the four target binaries or its signature is missing,
so a partial/unsigned release cannot be published.

## How to verify a release

Download a binary, its `.sig`, its `.pem`, and `SHA256SUMS` from the
[release page](https://github.com/SuarezPM/apohara-compliance/releases), then run any or
all of the three independent checks below. The example uses the linux target — substitute
your platform's asset name.

```bash
BIN=apohara-compliance-scanner-x86_64-unknown-linux-gnu
```

**1. Checksum** — confirms the bytes match what the release recorded:

```bash
grep "$BIN\$" SHA256SUMS | sha256sum -c
# => apohara-compliance-scanner-x86_64-unknown-linux-gnu: OK
```

**2. cosign signature** — confirms the binary was signed by *this repo's release workflow*
(keyless / OIDC). The `.pem` is base64-wrapped as published, so decode it first:

```bash
base64 -d "$BIN.pem" > "$BIN.crt"   # the published .pem is base64-encoded
cosign verify-blob "$BIN" \
  --certificate "$BIN.crt" \
  --signature "$BIN.sig" \
  --certificate-identity-regexp '^https://github\.com/SuarezPM/apohara-compliance/\.github/workflows/release\.yml@refs/tags/v.*$' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com
# => Verified OK
```

**3. Build provenance** — confirms GitHub's attestation that the binary was built by this
repo's workflow:

```bash
gh attestation verify "$BIN" --repo SuarezPM/apohara-compliance
# => ✓ Verification succeeded!
```

> The lowest-trust install path is `cargo install apohara-compliance-scanner`, which builds
> from the published crate source on your own machine. The signed binaries exist for users
> who prefer a prebuilt artifact with verifiable provenance.
